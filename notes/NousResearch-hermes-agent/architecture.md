## 一、整体架构

### 四层架构

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: Gateway (通信网关)                              │
│  统一入口，多通道接入                                      │
│  Telegram / Discord / Slack / WhatsApp / Signal / CLI    │
└───────────────────────────┬─────────────────────────────┘
                            │ 统一消息格式
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 2: Agent Core (智能核心)                           │
│  推理 + 学习 + 记忆 + 调度                               │
│  ┌─────┐ ┌─────────┐ ┌──────┐ ┌──────┐ ┌─────────┐    │
│  │推理  │ │学习循环  │ │记忆  │ │调度  │ │Subagent │    │
│  │Loop │ │CL Loop  │ │Mem  │ │Cron │ │Delegate│    │
│  └─────┘ └─────────┘ └──────┘ └──────┘ └─────────┘    │
└───────────────────────────┬─────────────────────────────┘
                            │ 执行指令
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Terminal Backends (执行层)                      │
│  7 种后端：执行代码、运行命令、管理环境                     │
│  Local / Docker / SSH / Singularity / Modal / Daytona    │
│  / Vercel Sandbox                                       │
└───────────────────────────┬─────────────────────────────┘
                            │ API 调用
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 4: Model Providers (模型层)                        │
│  10+ 提供商，200+ 模型                                   │
│  Nous Portal / OpenRouter / NVIDIA / HuggingFace / ...  │
└─────────────────────────────────────────────────────────┘
```

### 核心设计原则

| 原则 | 体现 |
|------|------|
| 自进化 | 闭环学习，使用中自动改进 |
| 模型无关 | 10+ 提供商，200+ 模型可切换 |
| 平台无关 | 6 种通信通道统一 |
| 环境无关 | 7 种终端后端可选 |
| 研究级 | 轨迹生成/压缩用于模型训练 |

## 二、闭环学习系统 (Closed Learning Loop)

### 学习循环架构

```
┌─────────────────────────────────────────────────────────┐
│                    使用中学习                              │
│                                                         │
│  对话/执行 → 观察模式 → 创建 Skill → 下次使用时改进       │
│      ↑                                    │             │
│      └────────────── 反馈 ────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

### 学习机制组件

| 组件 | 功能 |
|------|------|
| Skill 自创建 | 从经验中自动生成新 Skills |
| 使用中改进 | Skills 在使用过程中自动优化 |
| 知识持久化提示 | 提醒用户保存重要发现 |
| 会话搜索 (FTS5) | 全文搜索过去的对话 |
| 用户建模 (Honcho) | 辩证法用户模型 |

### 与 ECC Instincts 的对比

| 维度 | hermes-agent | ECC Instincts |
|------|-------------|---------------|
| 学习方式 | 闭环自进化 | 观察→评分→进化 |
| 存储 | Agent 策展 + FTS5 | SQLite + 项目隔离 |
| 进化路径 | Skill 自我改善 | Instinct → Skill/Agent |
| 用户建模 | Honcho 辩证法 | 无 |
| 研究支持 | 轨迹生成/压缩 | 无 |

## 三、通信网关 (Gateway)

### 单一网关架构

```
┌──────────┐ ┌──────────┐ ┌──────────┐
│ Telegram │ │ Discord  │ │  Slack   │
└────┬─────┘ └────┬─────┘ └────┬─────┘
     │             │             │
     └─────────────┼─────────────┘
                   │
                   ▼
┌─────────────────────────────────────┐
│        Gateway Process               │
│  统一消息格式化 + 路由 + 鉴权        │
└─────────────────┬───────────────────┘
```

## 四、Terminal Backends

### 7 种后端对比

| 后端 | 特点 | 适用场景 |
|------|------|---------|
| Local | 零延迟，完全控制 | 日常开发 |
| Docker | 隔离，可复现 | CI/CD，测试 |
| SSH | 远程服务器 | 生产环境 |
| Singularity | HPC 容器 | 科学计算 |
| Modal | 无服务器，按需计费 | 突发任务 |
| Daytona | 云开发环境 | 团队协作 |
| Vercel Sandbox | Web 沙箱 | 前端预览 |

### 无服务器持久化

```
Modal / Daytona 模式:

活跃时: 正常运行，按秒计费
空闲时: 自动休眠，零成本
重新激活: 下次请求自动唤醒

= "只在需要时付费"
```

## 六、Subagent 委派

```
主 Agent 决定需要并行处理
    │
    ├── Subagent A: 处理代码任务 (隔离上下文)
    ├── Subagent B: 搜索相关资料 (隔离上下文)
    └── Subagent C: 运行测试 (隔离上下文)
    │
    └── 结果汇总到主 Agent
```

## 七、研究功能

### 轨迹生成与压缩

```
目的: 为模型训练生成高质量数据

batch 模式:
  输入任务列表 → Agent 执行 → 记录完整轨迹 → 输出训练数据

压缩模式:
  原始轨迹 (长) → 压缩算法 → 精华轨迹 (短)
  = 保留关键决策，去除冗余步骤
  = 用于高效模型微调
```
