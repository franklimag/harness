# 工作流技能学习笔记

> to-prd、to-issues、triage、handoff、prototype — 管理完整开发工作流的技能族。

## 系统概述

工作流技能覆盖了从需求到交付的完整开发生命周期管理。它们不直接产出代码，而是管理"做什么"和"怎么组织做"。

### 技能族全景

| Skill | 阶段 | 输入 | 输出 |
|-------|------|------|------|
| to-prd | 需求阶段 | 模糊想法/需求 | 结构化 PRD 文档 |
| to-issues | 规划阶段 | PRD 文档 | 可执行的 Issue 列表 |
| triage | 维护阶段 | Bug 报告/Feature Request | 优先级排序 |
| handoff | 交接阶段 | 当前工作状态 | 交接文档 |
| prototype | 验证阶段 | 设计想法 | 可运行原型 |

### 完整工作流链路

```
模糊想法
    │
    ▼
┌──────────┐     ┌───────────┐     ┌─────────┐
│  to-prd  │────▶│ to-issues │────▶│   tdd   │
│ 需求文档化│     │ Issue 拆分 │     │ 编码实现 │
└──────────┘     └───────────┘     └─────────┘
                                        │
    ┌───────────────────────────────────┘
    ▼
┌──────────┐     ┌───────────┐
│ prototype│     │  handoff  │
│ 快速验证  │     │ 工作交接  │
└──────────┘     └───────────┘
```

## to-prd 技能详解

### 从模糊到清晰

```
输入 (模糊):
  "我们需要一个通知系统"

输出 (PRD):
  ## 通知系统 PRD
  
  ### 目标
  用户能实时收到相关事件通知
  
  ### 用户故事
  - 作为用户，我想收到新消息通知
  - 作为管理员，我想配置通知渠道
  
  ### 技术约束
  - 延迟 < 1s
  - 支持 email + push + in-app
  
  ### 不做什么
  - 不做 SMS（成本原因）
  - 不做实时视频通知
  
  ### 验收标准
  - [ ] 新消息 1 秒内推送
  - [ ] 用户可关闭特定类型通知
  - [ ] 管理员可配置全局通知策略
```

### PRD 结构

| 章节 | 内容 | 为什么需要 |
|------|------|-----------|
| 目标 | 一句话说明 | 对齐方向 |
| 用户故事 | 用户视角的需求 | 验证完整性 |
| 技术约束 | 性能/安全/兼容 | 避免返工 |
| 不做什么 | 明确排除项 | 控制范围 |
| 验收标准 | 可检验的完成条件 | 定义 "done" |

### to-prd 与 grill-me 的配合

```
grill-me: "你说的通知系统，具体指什么？"
用户: "就是消息来了给用户提醒"
grill-me: "只是 push 还是也要 email？"
用户: "都要，但不要 SMS"
    │
    ▼
to-prd: 将以上对话整理为结构化 PRD
```

## to-issues 技能详解

### 从 PRD 到可执行 Issues

```
PRD (通知系统)
    │
    ▼
to-issues 拆分:

Issue #1: 设计通知数据模型
  - 优先级: P0
  - 估时: 2h
  - 标签: backend, database

Issue #2: 实现推送通知服务
  - 优先级: P0
  - 估时: 4h
  - 标签: backend, notification
  - 依赖: #1

Issue #3: 实现邮件通知服务
  - 优先级: P1
  - 估时: 3h
  - 标签: backend, email
  - 依赖: #1

Issue #4: 用户通知偏好设置 UI
  - 优先级: P1
  - 估时: 4h
  - 标签: frontend, settings
  - 依赖: #1

Issue #5: 管理员通知策略配置
  - 优先级: P2
  - 估时: 3h
  - 标签: admin, settings
```

### Issue 拆分原则

| 原则 | 说明 | 反例 |
|------|------|------|
| 独立可交付 | 每个 Issue 可独立完成和验证 | "实现前后端" (太大) |
| 明确边界 | 清楚什么做什么不做 | "优化性能" (太模糊) |
| 可估时 | 2-8 小时范围 | "重构整个系统" (无法估时) |
| 有验收标准 | 明确怎样算完成 | "改善体验" (无法验证) |
| 标注依赖 | 说明前置条件 | 忽略依赖导致阻塞 |

## triage 技能详解

### Bug/Feature 分类矩阵

```
         影响大
           │
     P0    │    P1
  (紧急修复) │  (下个迭代)
           │
  ─────────┼─────────── 频率高/低
           │
     P2    │    P3
  (排期修复) │  (有空再说)
           │
         影响小
```

### Triage 决策流程

```
新报告进入
    │
    ├── 是 Bug 还是 Feature?
    │       │
    │       ├── Bug:
    │       │    ├── 影响生产? → P0 (立即修)
    │       │    ├── 影响体验? → P1 (本周修)
    │       │    └── 边缘情况? → P2 (排期修)
    │       │
    │       └── Feature:
    │            ├── 用户强需? → P1 (规划中)
    │            ├── 锦上添花? → P2 (积压)
    │            └── 个人偏好? → P3 (观察)
    │
    └── 标签: bug/feature/enhancement + priority + component
```

## handoff 技能详解

### 解决 Agent 的"金鱼记忆"

```
问题:
  Session 1: Agent 理解了上下文，做了一半工作
  Session 2: Agent 完全忘记了 Session 1 的一切
  = 从头开始，浪费时间

解法 (handoff):
  Session 1 结束前: Agent 生成 handoff 文档
  Session 2 开始时: Agent 读取 handoff 文档
  = 无缝继续
```

### Handoff 文档结构

```markdown
# Handoff — [任务名]

## 当前状态
- [x] 数据模型设计完成
- [x] API 框架搭建完成
- [ ] 业务逻辑实现 (进行中)
- [ ] 测试编写 (未开始)

## 关键决策
1. 使用 PostgreSQL 而非 MongoDB (原因: 关系型数据)
2. API 使用 RESTful 而非 GraphQL (原因: 简单场景)

## 当前卡点
- `UserService.createUser()` 需要处理邮箱去重

## 下次继续
1. 完成 createUser 的去重逻辑
2. 写对应的测试
3. 开始下一个 Issue

## 重要文件
- src/services/user.ts — 主要修改文件
- src/models/user.ts — 数据模型
- tests/user.test.ts — 测试文件
```

### Handoff 的使用场景

| 场景 | handoff 的价值 |
|------|---------------|
| 超时中断 | 长任务做不完，记录进度 |
| 换模型/工具 | 从 Claude Code 到 Cursor |
| 团队协作 | 一人做一半，另一人接手 |
| 知识保存 | 记录决策原因，避免将来重复思考 |

## prototype 技能详解

### 快速验证而非完美实现

```
prototype 的原则:

✅ 做:
- 核心流程可运行
- 关键交互可体验
- 技术可行性验证

❌ 不做:
- 错误处理
- 边界情况
- 性能优化
- 代码质量
- 完整测试
```

### Prototype 使用时机

| 时机 | 目的 | 产出 |
|------|------|------|
| 技术选型 | 验证方案是否可行 | 概念验证代码 |
| 设计验证 | 验证交互是否合理 | 可点击原型 |
| 需求确认 | 让用户看到"长什么样" | 演示版本 |
| 架构验证 | 验证架构是否支撑需求 | 骨架代码 |

### Prototype → 正式代码 的过渡

```
prototype (验证成功)
    │
    ├── 不要: 直接在 prototype 上继续写
    │         (会变成技术债)
    │
    └── 要: 用 prototype 的知识指导正式开发
            1. to-prd (基于 prototype 经验写 PRD)
            2. to-issues (拆分任务)
            3. tdd (正式实现)
            4. 丢弃 prototype 代码
```

## 完整工作流示例

```
Day 1: 需求阶段
  用户: "想加个搜索功能"
  │
  ├── grill-me: 确认搜索范围、对象、UI 期望
  └── to-prd: 输出搜索功能 PRD

Day 2: 规划阶段
  │
  ├── to-issues: PRD → 5 个 Issues
  └── triage: 确定优先级 (P0: 全文搜索, P1: 过滤, P2: 高亮)

Day 3: 验证阶段
  │
  └── prototype: 快速验证搜索库选型 (Algolia vs Elasticsearch)

Day 4-6: 实现阶段
  │
  ├── tdd: 按 Issue 顺序实现
  └── diagnose: 解决实现中遇到的问题

Day 6: 交接
  │
  └── handoff: 生成交接文档，供下次 session 继续
```
