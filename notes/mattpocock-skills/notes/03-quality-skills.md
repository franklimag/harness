# 质量技能学习笔记

> tdd、diagnose、improve-codebase-architecture — 保障代码质量和架构健康的核心技能族。

## 系统概述

质量技能解决 mattpocock/skills 的两大核心问题："代码不能用"和"代码变成泥球"。通过 TDD 保障正确性，通过 diagnose 定位问题，通过架构改善防止劣化。

### 技能族全景

| Skill | 目标 | 使用阶段 |
|-------|------|---------|
| tdd | 代码正确性保障 | 编码中 |
| diagnose | 问题系统化定位 | 出问题时 |
| improve-codebase-architecture | 架构持续改善 | 定期维护 |
| zoom-out | 全局视角审视 | 架构决策前 |

## TDD 技能详解

### Red-Green-Refactor 循环

```
┌──────────────────────────────────────────────┐
│                                              │
│    ┌─────┐    ┌─────────┐    ┌──────────┐   │
│    │ Red │───▶│  Green  │───▶│ Refactor │   │
│    │写失败│    │最小通过  │    │安全重构   │   │
│    └─────┘    └─────────┘    └────┬─────┘   │
│       ▲                           │          │
│       └───────────────────────────┘          │
│              下一个行为                        │
└──────────────────────────────────────────────┘
```

### TDD 的反直觉之处

| 直觉 | TDD 实践 | 原因 |
|------|---------|------|
| "先想好再写" | 先写测试再想实现 | 测试驱动设计 |
| "写完美代码" | 先写丑代码通过测试 | 过早优化是万恶之源 |
| "一次写完" | 一次只加一个行为 | 小步前进，减少风险 |
| "测试是额外工作" | 测试是设计工具 | 测试即文档即验证 |

### TDD 步骤拆解

#### Step 1: Red（写失败测试）

```
输入: 需求描述
输出: 一个失败的测试

要点:
├── 测试名描述"期望行为"
├── 先写断言 (expect/assert)
├── 再写 setup
├── 运行确认是"因功能未实现"而失败
└── 不是因为语法错误或导入缺失
```

#### Step 2: Green（最小实现）

```
输入: 失败的测试
输出: 通过的测试 + 最少代码

要点:
├── 写最少代码让测试通过
├── 可以硬编码返回值（如果只有一个测试）
├── 不做"预见性"设计
├── 不处理"将来可能需要的"情况
└── 丑代码是可以的，重构阶段处理
```

#### Step 3: Refactor（安全重构）

```
输入: 通过的测试 + 可能较丑的代码
输出: 干净的代码 + 仍通过的测试

要点:
├── 消除重复 (DRY)
├── 改善命名
├── 提取函数/方法
├── 简化条件逻辑
├── 每次小修改后运行测试
└── 不添加新行为（那是下一个 Red）
```

### TDD 在 Agent 协作中的价值

```
无 TDD 的 Agent:
  编写功能 → 可能有 bug → 用户发现 → Agent 修复 → 引入新 bug → 循环
  = 不可预测的质量

有 TDD 的 Agent:
  写测试 → 实现 → 测试通过 → 重构 → 测试通过 → 交付
  = 可验证的质量

关键差异: 测试是"自动化的用户反馈"
```

## Diagnose 技能详解

### 系统化诊断 vs "试试看"

```
┌─────────────────────────────────────────────┐
│  ❌ "试试看"模式:                            │
│                                             │
│  看到错误 → 猜一个原因 → 改代码 → 不行      │
│          → 猜另一个 → 改 → 不行             │
│          → 继续猜... → 也许能修好            │
│                                             │
│  问题: 浪费时间，可能引入新问题              │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  ✅ diagnose 系统化模式:                     │
│                                             │
│  1. 观察: 收集所有症状和错误信息             │
│  2. 假设: 列出可能的根因（按概率排序）       │
│  3. 验证: 设计最小实验验证/排除假设          │
│  4. 定位: 确认根因                           │
│  5. 修复: 针对性修复 + 回归测试              │
│                                             │
│  优势: 高效、可靠、不引入新问题              │
└─────────────────────────────────────────────┘
```

### Diagnose 五步法

| 步骤 | 行动 | 产出 |
|------|------|------|
| 1. 观察 | 收集错误信息、日志、复现步骤 | 症状清单 |
| 2. 假设 | 列出可能原因，按概率排序 | 假设列表 |
| 3. 验证 | 设计最小实验验证假设 | 验证结果 |
| 4. 定位 | 确认根因，排除干扰 | 根因确认 |
| 5. 修复 | 针对性修复 + 验证 | 修复完成 |

### Diagnose 与 TDD 的协同

```
Bug 报告 → diagnose (定位根因)
              │
              ▼
         tdd (Red: 写一个复现 bug 的测试)
              │
              ▼
         tdd (Green: 修复 bug 让测试通过)
              │
              ▼
         tdd (Refactor: 清理修复代码)
              │
              ▼
         永远不会再出现这个 bug (回归测试保护)
```

## Improve-Codebase-Architecture 技能详解

### 架构劣化模式

```
时间 →
┌────────────────────────────────────────────────┐
│                                                │
│  优雅架构                                       │
│    \                                           │
│     \   "快速添加功能"                          │
│      \    ↓                                    │
│       \──────── 技术债累积                      │
│               \                                │
│                \──── "大泥球" (Ball of Mud)     │
│                                                │
│  ── 没有 improve-codebase-architecture ──      │
└────────────────────────────────────────────────┘

┌────────────────────────────────────────────────┐
│                                                │
│  优雅架构                                       │
│    \  /\  /\  /\  /\                           │
│     \/  \/  \/  \/  \──── 持续健康              │
│                                                │
│  加功能 → 小劣化 → 改善 → 加功能 → 小劣化 →改善│
│                                                │
│  ── 有 improve-codebase-architecture ──        │
└────────────────────────────────────────────────┘
```

### 架构改善策略

| 策略 | 时机 | 方法 |
|------|------|------|
| 增量改善 | 每次提交 | "Boy Scout Rule" — 离开时比到达时更干净 |
| 定期审视 | 每周/每迭代 | 用 zoom-out 看全局，识别劣化趋势 |
| 关键重构 | 达到阈值时 | 大规模结构调整（需要 to-prd + to-issues） |

### Zoom-Out 全局视角

zoom-out Skill 让 Agent 跳出当前文件/模块，审视系统全貌：

```
当前视角 (编码时):
┌─────────┐
│ 一个文件 │  ← Agent 通常只看这里
└─────────┘

zoom-out 视角:
┌─────────────────────────────────────────┐
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐   │
│  │模块A│──│模块B│──│模块C│──│模块D│   │
│  └─────┘  └─────┘  └─────┘  └─────┘   │
│       │                   │             │
│       └───────────────────┘             │
│         "这个耦合是否必要？"              │
└─────────────────────────────────────────┘
```

### zoom-out 的触发时机

| 信号 | 说明 |
|------|------|
| 修改一处影响多处 | 耦合过紧 |
| 同样的模式重复出现 | 缺少抽象 |
| 新功能不知道放哪里 | 边界不清 |
| 理解代码需要读多个文件 | 组织混乱 |
| 测试需要大量 mock | 依赖过多 |

## 三个质量技能的组合工作流

```
日常开发:
  tdd ──── 保障每次修改的正确性

出问题时:
  diagnose ──── 系统化定位根因
      │
      └── tdd ──── 用测试固化修复

定期维护:
  zoom-out ──── 审视架构健康度
      │
      └── improve-codebase-architecture ──── 增量改善
              │
              └── tdd ──── 重构时保障不引入回归
```
