# SKILL.md 格式与渐进式披露模式

> mattpocock/skills 的核心设计 — 纯 Markdown 指令文件的格式规范与信息架构。

## 概述

mattpocock/skills 中每个 Skill 都是一个独立的 `SKILL.md` 文件。与 ECC 的 YAML frontmatter 格式不同，这里采用**纯 Markdown**，无任何元数据头，无运行时依赖，无解析要求。

### 核心特征

| 特征 | 说明 |
|------|------|
| 格式 | 纯 Markdown（无 YAML frontmatter） |
| 依赖 | 无（不需要解析器/运行时） |
| 大小 | 通常 50-200 行 |
| 可读性 | 人类和 Agent 都可直接阅读 |
| 修改 | 直接编辑文本，无 schema 约束 |

## 渐进式披露 (Progressive Disclosure)

### 核心理念

渐进式披露是 mattpocock/skills 最重要的设计模式。信息不一次性全部呈现，而是按层次逐步展开：

```
┌─────────────────────────────────────────────────────┐
│  Level 1: 标题 + 一句话描述                          │
│  ─── Agent 快速判断: "这个 Skill 是否适用当前场景？"  │
└────────────────────────┬────────────────────────────┘
                         │ 适用 → 继续读
                         ▼
┌─────────────────────────────────────────────────────┐
│  Level 2: 触发条件 + 基本步骤                        │
│  ─── Agent 决定: "如何开始执行？"                    │
└────────────────────────┬────────────────────────────┘
                         │ 执行中 → 按需深入
                         ▼
┌─────────────────────────────────────────────────────┐
│  Level 3: 详细指导 + 边界情况                        │
│  ─── Agent 处理: "遇到特殊情况怎么办？"              │
└────────────────────────┬────────────────────────────┘
                         │ 需要示例 → 最后一层
                         ▼
┌─────────────────────────────────────────────────────┐
│  Level 4: 完整示例 + 反模式                          │
│  ─── Agent 参考: "好的做法 vs 坏的做法"              │
└─────────────────────────────────────────────────────┘
```

### 为什么需要渐进式披露

| 问题 | 非渐进式 | 渐进式 |
|------|---------|--------|
| Token 消耗 | 每次加载全部内容 | 按需加载相关层级 |
| 信息过载 | Agent 被海量指令淹没 | Agent 逐步获取所需信息 |
| 相关性 | 大量不相关内容 | 每层都高度相关 |
| 决策速度 | 需要消化全部才能行动 | Level 1-2 即可开始 |

### 认知负载理论基础

渐进式披露源自 John Sweller 的认知负载理论：

```
工作记忆容量有限 → 一次只能处理有限信息
                     │
                     ├── 内在负载 (任务本身的复杂度) → 不可减少
                     ├── 外在负载 (表达方式的复杂度) → 可以减少 ✓
                     └── 关联负载 (整合新旧知识) → 应该增加
                     
渐进式披露的作用: 减少外在负载，增加关联负载
```

### 实际示例: tdd Skill

```markdown
# TDD (Test-Driven Development)

> 编写代码前先写测试。通过 Red-Green-Refactor 循环确保代码正确性。

## 何时使用
- 实现新功能
- 修复已知 Bug
- 重构现有代码
- 添加边界条件处理

## 流程

### 1. Red — 编写失败测试
1. 分析需求，确定预期行为
2. 编写最小测试用例描述该行为
3. 运行测试，确认失败（红色）
4. 失败原因必须是"功能未实现"，不是测试写错

### 2. Green — 最小实现
1. 写最少的代码让测试通过
2. 不做任何"预见性"设计
3. 可以写"丑"代码，只要测试通过
4. 运行测试，确认通过（绿色）

### 3. Refactor — 安全重构
1. 在测试保护下重构代码
2. 消除重复 (DRY)
3. 改善命名和结构
4. 每次修改后运行测试确认仍绿
5. 不添加新功能

## 关键原则
- 每次只测一个行为
- 测试名描述"期望行为"，不描述"实现细节"
- 先写断言 (assert)，再写 setup
- 测试之间互相独立，无共享状态
- Mock 数量越少越好

## 边界情况
- 测试不知道怎么写: 先用 grill-me 确认需求
- 测试通过但感觉不对: 添加更多边界用例
- 重构改变了太多: 回退，分成更小的步骤
- 遗留代码无测试: 先加 characterization test

## 示例

### 好的做法
```
// 测试名清楚描述行为
it('returns empty array when no items match filter', () => {
  const items = [{ name: 'a', active: false }];
  expect(filterActive(items)).toEqual([]);
});
```

### 避免的做法
```
// 测试名描述实现而非行为
it('calls array.filter method', () => { ... });

// 测试多个行为
it('filters and sorts and paginates', () => { ... });

## 格式设计决策

### 决策 1: 为什么不用 YAML Frontmatter

| 有 YAML | 无 YAML |
|---------|---------|
| 需要解析器 | 纯文本即可 |
| 格式错误导致失败 | 不可能格式错误 |
| 修改需了解 schema | 随意修改 |
| Agent 需要处理两种格式 | Agent 只处理 Markdown |
| 支持机器读取元数据 | 人类优先，机器适应 |

**结论**: Skills 的核心价值是"指导行为"而非"声明元数据"，YAML 是不必要的复杂度。

### 决策 2: 为什么不分章节文件

```
方案 A (分文件):
tdd/
├── when-to-use.md
├── steps.md
├── principles.md
└── examples.md

方案 B (单文件) ← 选择这个:
tdd/
└── SKILL.md (包含所有内容)
```

**原因**:
- Agent 一次 `read_file` 获取全部内容
- 无需多次文件读取消耗 token
- 上下文连贯，不在文件间跳转
- 渐进式披露在文件内通过标题层级实现

### 决策 3: 文件命名规范

| 组件 | 规范 | 示例 |
|------|------|------|
| 目录名 | kebab-case | `grill-with-docs/` |
| 文件名 | 固定为 SKILL.md | `SKILL.md` |
| 标题 | 自然语言 | `# Grill With Docs` |

## write-a-skill 元技能

`write-a-skill` 是一个"写 Skill 的 Skill"，提供了编写新 Skill 的模板和指导：

### 编写新 Skill 的流程

```
1. 确定行为模式
   └── "我想教 Agent 什么行为？"

2. 选择渐进式披露层级
   └── Level 1-4 分别放什么？

3. 编写 SKILL.md
   └── 按模板填写

4. 测试有效性
   └── 让 Agent 使用，观察效果

5. 迭代优化
   └── 根据 Agent 行为调整措辞
```

### 编写好 Skill 的原则

| 原则 | 说明 |
|------|------|
| 具体 > 抽象 | "运行 npm test" > "执行测试" |
| 命令式 > 描述式 | "先写测试" > "通常应该先写测试" |
| 步骤明确 | 1-2-3 编号步骤，不模糊 |
| 包含"不做什么" | 反模式同样重要 |
| 短句优先 | Agent 理解短句更好 |
| 无歧义 | 避免"适当"、"可能"、"一般来说" |
