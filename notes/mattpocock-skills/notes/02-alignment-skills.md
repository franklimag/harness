# 对齐技能学习笔记

> grill-me、grill-with-docs、CONTEXT.md — 确保 Agent 理解与用户一致的核心技能族。

## 系统概述

对齐技能是 mattpocock/skills 中解决"Agent 不做我想要的事"这一核心问题的 Skill 族。它们的共同目标是：**在编码前确保 Agent 真正理解用户意图**。

### 核心数据

| Skill | 解决的问题 | 使用时机 |
|-------|-----------|---------|
| grill-me | Agent 不确认就动手 | 编码前 |
| grill-with-docs | Agent 对文档理解有误 | 涉及已有文档时 |
| CONTEXT.md | Agent 不懂项目特定术语 | 项目初始化时创建 |

### 为什么需要对齐

```
常见失败模式:

用户: "帮我加个用户系统"
Agent: (直接开始写一个完整的认证系统)
用户: "不是...我只是想要一个简单的用户列表页面"
Agent: (推翻重来)
用户: "......"

─── 浪费的资源: Token + 时间 + 用户耐心 ───

有 grill-me 后:

用户: "帮我加个用户系统"
Agent: "在我开始前，让我确认几个问题:
        1. '用户系统'指的是认证登录，还是用户信息展示？
        2. 需要注册功能还是只有管理员创建？
        3. 有现有的用户数据模型吗？"
用户: "只是一个列表页，显示已有用户"
Agent: (精准实现)

─── 节省: 2 分钟确认 = 省 20 分钟返工 ───
```

## grill-me 技能详解

### 核心机制

grill-me 的本质是**在 Agent 开始编码前插入一个结构化的确认环节**。

```
用户给出需求
    │
    ▼
┌───────────────────────────────────────┐
│  grill-me 介入                         │
│                                       │
│  1. Agent 不编码                       │
│  2. Agent 提出确认问题                  │
│  3. 用户回答                           │
│  4. Agent 复述理解                     │
│  5. 用户确认 "对" 或 "不对, 是..."     │
│  6. 循环直到对齐完成                    │
└───────────────────────────────────────┘
    │
    ▼
Agent 开始编码（已对齐）
```

### grill-me 提问策略

| 问题类型 | 目的 | 示例 |
|---------|------|------|
| 范围确认 | 明确边界 | "只处理 A 情况，还是也包括 B？" |
| 术语澄清 | 消除歧义 | "'部署'指的是 staging 还是 production？" |
| 约束条件 | 了解限制 | "有性能要求吗？需要支持多少用户？" |
| 已有资源 | 避免重复 | "有现成的组件/库可以用吗？" |
| 优先级 | 确定顺序 | "先实现核心功能还是先处理错误情况？" |
| 验证标准 | 定义完成 | "怎样算'做完了'？需要测试吗？" |

### grill-me 使用要点

```markdown
## 使用 grill-me 的正确方式

✅ 好的触发:
- "帮我重构这个模块" → Agent 先问 "重构的目标是什么？性能/可读性/可扩展性？"
- "加个缓存" → Agent 先问 "缓存什么数据？失效策略是什么？"
- "修这个 bug" → Agent 先问 "能描述复现步骤吗？期望行为是什么？"

❌ 不需要 grill-me 的情况:
- 极其简单明确的任务: "给这个变量改个名"
- 用户已经给了完整规格: "按照这个 spec 实现"
- 紧急修复已知问题: "线上这个报错修一下"
```

### grill-me 问题数量控制

| 任务复杂度 | 建议问题数 | 说明 |
|-----------|-----------|------|
| 简单 (改名/移动) | 0-1 | 可能不需要 grill |
| 中等 (新功能) | 2-4 | 核心问题即可 |
| 复杂 (重构/新系统) | 4-7 | 深入确认 |
| 特大 (架构变更) | 分批 | 先大方向，再细节 |

## grill-with-docs 技能详解

### 与 grill-me 的区别

| 维度 | grill-me | grill-with-docs |
|------|---------|----------------|
| 信息来源 | 用户口头描述 | 已有文档 |
| 对齐对象 | Agent 理解 vs 用户意图 | Agent 理解 vs 文档内容 |
| 使用场景 | 无现有文档时 | 有 API 文档/设计文档时 |
| 问题焦点 | "你想要什么" | "文档说的是这个意思吗" |

### grill-with-docs 工作流

```
用户: "按照 API 文档实现这个接口"
    │
    ▼
┌───────────────────────────────────────────────┐
│  grill-with-docs 介入                          │
│                                               │
│  1. Agent 读取相关文档                          │
│  2. Agent 列出自己的理解                        │
│  3. Agent 指出不确定/模糊的部分                  │
│  4. 用户确认或修正                              │
│  5. Agent 按确认的理解实现                      │
└───────────────────────────────────────────────┘
```

### 适用场景

| 场景 | 文档类型 | grill-with-docs 的价值 |
|------|---------|---------------------|
| API 实现 | OpenAPI/Swagger | 确认参数语义、错误码含义 |
| 设计实现 | Figma/设计稿 | 确认交互细节、边界情况 |
| 迁移工作 | 迁移指南 | 确认版本差异、兼容策略 |
| 集成开发 | 第三方文档 | 确认认证方式、限速规则 |

## CONTEXT.md 共享语言

### 核心理念

CONTEXT.md 是一个放在项目根目录的文件，定义了项目特有的术语和约定。它让 Agent 使用与团队一致的"语言"进行沟通。

### 设计灵感

来自 Eric Evans 的《领域驱动设计》中的"统一语言 (Ubiquitous Language)"：

```
DDD 统一语言:
  团队所有人（开发/产品/设计/Agent）
  使用同一套术语
  指代同一个概念
  = 消除沟通中的歧义
```

### CONTEXT.md 标准结构

```markdown
# CONTEXT.md

## 项目概述
[一句话描述项目做什么]

## 领域术语

| 术语 | 定义 | 注意事项 |
|------|------|---------|
| 用户 | 使用产品的最终消费者 | 不是管理员 |
| 客户 | 付费的企业实体 | 一个客户下有多个用户 |
| 工单 | 客户提交的支持请求 | 不是开发任务 |
| 租户 | 数据隔离的组织单位 | = 客户 |

## 架构术语

| 组件 | 含义 | 技术栈 |
|------|------|--------|
| 网关 | API 入口层 | Kong + Express |
| 核心 | 业务逻辑层 | TypeScript + NestJS |
| 存储 | 数据持久化 | PostgreSQL + Redis |

## 代码约定
- 分支: feature/xxx, fix/xxx, chore/xxx
- 提交: conventional commits
- 测试: 每个 PR 必须有测试
- Review: 至少 1 人 approve

## 不做什么
- 不直接操作数据库（通过 ORM）
- 不在 controller 写业务逻辑
- 不使用 any 类型
```

### CONTEXT.md 与 grill-me 的协同

```
没有 CONTEXT.md:
  用户: "把工单关了"
  Agent: "你是说 GitHub Issue 吗？"  ← 歧义
  用户: "不是，是客服工单"
  Agent: "哦，是 Zendesk ticket？"   ← 还在猜

有 CONTEXT.md:
  Agent 已知: 工单 = 客户提交的支持请求 (Zendesk)
  用户: "把工单关了"
  Agent: (直接操作 Zendesk API 关闭工单)  ← 零歧义
```

### CONTEXT.md 维护策略

| 时机 | 行动 |
|------|------|
| 项目初始化 | 创建基础 CONTEXT.md |
| 新团队成员加入 | 补充常见误解 |
| Agent 理解错误时 | 添加澄清条目 |
| 架构变更时 | 更新架构术语 |
| 定期回顾 | 删除过时条目 |

## 三个对齐技能的组合使用

### 推荐组合策略

```
场景 1: 全新功能开发
├── CONTEXT.md (已存在) → 提供背景知识
├── grill-me → 确认功能需求细节
└── 开始编码

场景 2: 基于文档的实现
├── CONTEXT.md (已存在) → 提供背景知识
├── grill-with-docs → 确认文档理解
└── 开始编码

场景 3: 全新项目
├── 创建 CONTEXT.md → 建立共享语言
├── grill-me → 确认项目目标
└── 开始规划和编码
```

### 对齐效果量化

| 指标 | 无对齐 | 有对齐 | 改善 |
|------|--------|--------|------|
| 首次实现准确率 | ~40% | ~85% | +45% |
| 返工次数 | 2-4 次 | 0-1 次 | -75% |
| Token 浪费 | 高 | 低 | -60% |
| 用户满意度 | 低 | 高 | 显著提升 |
