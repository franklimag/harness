# Subagent-Driven-Development 学习笔记

> superpowers 的实现阶段核心模式 — 子 Agent 独立执行任务，主 Agent 协调审查。

## 概述

Subagent-Driven-Development (子 Agent 驱动开发) 是 superpowers 在 Implementation 阶段的核心执行模式。主 Agent 作为协调者，将 Plan 中的任务分配给独立的 Subagent 执行。

### 核心架构

```
┌───────────────────────────────────────────────────┐
│              主 Agent (Orchestrator)                │
│                                                   │
│  职责:                                            │
│  - 读取 Plan                                     │
│  - 分配任务                                       │
│  - 协调依赖                                       │
│  - 审查产出                                       │
│  - 整合结果                                       │
└────────────┬──────────────┬───────────────────────┘
             │              │              │
             ▼              ▼              ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Subagent 1   │ │ Subagent 2   │ │ Subagent 3   │
│ Task: 数据模型│ │ Task: 服务层 │ │ Task: API 层 │
│ 独立上下文    │ │ 独立上下文    │ │ 独立上下文    │
│ 遵循 TDD     │ │ 遵循 TDD     │ │ 遵循 TDD     │
└──────────────┘ └──────────────┘ └──────────────┘
```

## 为什么需要 Subagent

### 单一 Agent 的问题

| 问题 | 说明 |
|------|------|
| 上下文溢出 | 长任务导致上下文窗口耗尽 |
| 注意力分散 | 处理多个模块时质量下降 |
| 失败扩散 | 一个错误影响后续所有工作 |
| 不可审查 | 大块产出难以逐步验证 |

### Subagent 的解决方案

| 解决方式 | 说明 |
|---------|------|
| 上下文隔离 | 每个 Subagent 只需要自己任务的上下文 |
| 专注单一 | 一个 Subagent 只做一件事 |
| 失败隔离 | 一个失败不影响其他 |
| 易于审查 | 小块产出逐个检查 |

## Subagent 生命周期

```
┌────────────────────────────────────────────────┐
│  1. 创建 (Create)                              │
│     - 主 Agent 确定需要 Subagent               │
│     - 给出: 任务描述 + 约束条件 + 验证标准     │
│     - 提供: 必要的文件列表和上下文              │
└────────────────────┬───────────────────────────┘
                     ▼
┌────────────────────────────────────────────────┐
│  2. 执行 (Execute)                             │
│     - Subagent 独立工作                        │
│     - 遵循 TDD (先测试后实现)                  │
│     - 只在分配的范围内工作                      │
│     - 完成后报告结果和产出                      │
└────────────────────┬───────────────────────────┘
                     ▼
┌────────────────────────────────────────────────┐
│  3. 审查 (Review)                              │
│     - 主 Agent 检查 Subagent 产出              │
│     - 代码质量/测试通过/符合计划                │
│     - 通过 → 整合; 不通过 → 修正指令           │
└────────────────────┬───────────────────────────┘
                     ▼
┌────────────────────────────────────────────────┐
│  4. 整合 (Integrate)                           │
│     - 将 Subagent 产出合并到主代码库           │
│     - 确认整合后测试仍通过                      │
│     - 更新进度追踪                              │
└────────────────────────────────────────────────┘
```

## 任务分配策略

### 什么适合分配给 Subagent

| 适合 | 不适合 |
|------|--------|
| 有明确输入输出的任务 | 模糊的探索性工作 |
| 可独立验证的工作 | 需要频繁与用户交互的 |
| 代码实现类 | 架构决策类 |
| 有测试可验证的 | 需要主观判断的 |

### 任务描述模板

```markdown
## Subagent Task: [任务名]

### 目标
[一句话说明要完成什么]

### 输入
- 依赖文件: [列表]
- 前置 Subagent 产出: [如有]

### 约束
- 遵循 TDD
- 不修改 [列出不能动的文件]
- 使用 [指定技术/模式]

### 验证标准
- [ ] 测试通过
- [ ] 符合 [具体标准]
- [ ] 不超过 [行数/复杂度] 限制

### 产出
- 新建/修改的文件列表
```

## Subagent 与 TDD 的结合

```
每个 Subagent 的工作流:

  接收任务
      │
      ▼
  Red: 编写测试 (描述期望行为)
      │
      ▼
  Green: 实现代码 (最小通过)
      │
      ▼
  Refactor: 优化 (保持测试通过)
      │
      ▼
  报告: 产出文件 + 测试结果
```

## 依赖协调

### 串行依赖

```
Task 1 (数据模型) → Task 2 (服务层) → Task 3 (API 层)

处理方式:
  Subagent 1 完成 → 主 Agent 审查通过 → Subagent 2 开始
```

### 并行可能

```
Task A (用户模块) ─┐
                   ├── Task C (集成测试) 
Task B (通知模块) ─┘

处理方式:
  Subagent A 和 Subagent B 可并行执行
  都完成后 → Subagent C 执行集成
```

## 与其他系统的 Subagent 对比

| 维度 | superpowers | hermes-agent | ECC |
|------|------------|--------------|-----|
| Subagent 触发 | Plan 驱动 | 手动/自动 | 无（单 Agent） |
| 通信方式 | 文件产出 | 进程通信 | — |
| 审查机制 | 主 Agent Review | 自动验证 | — |
| TDD 强制 | 是 | 可选 | 可选 |
| 上下文隔离 | 完全隔离 | 部分隔离 | — |
